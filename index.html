Contour

is a different take on CSS libraries

Foreword

Contour was conceived to be an alternative to both utility-first and component based libraries. It aims to provide flexibility to the UI implementors while maintaining a clear separation between style and content.

Instead of providing the a set of opinionated implementation or atomic classes, Contour provide a flexible foundation that allows rich customizability and a list of reference Variants based on this foundation. It then makes this foundation scalable by using CSS layers to manage specificity and CSS custom properties to reinforce consistency.

Convention

Contour is as much about the implemntation as it is about setting up a convention for creating styling system. Any styling system, no matter how well designed, requires certain discipline, structure, and mutual understanding between members of the group using the system.

BEM but different

On the surface, the naming of classes follows the BEM convetion. In Contour, however, the convetion is given new meaning:

component__element--variation

Component are the root-level reusable parts such as buttons and notices. Elements are parts that make up these components. Like BEM, elements can only exist within its component. In other word, you cannot use button__section within notice

Manage Specificity

We limit the specificity elevation by doing 2 things:

  1. keeping things that are "custom" and can get messy in a higher layer, such as @layer variant.
  2. use child and nested selectors only in higher layers, not base For example, if you want a universal variant of a base component or a element, do so in the variant layer.

If you want to override things in the variant layer for a specific page, simply create a new layer called page and do it there.

Custom Property

One of the most defining feature of Contour is its liberal use of CSS custom properties. Custom properties, like many properties such as color for font-size, are inherited by default.
